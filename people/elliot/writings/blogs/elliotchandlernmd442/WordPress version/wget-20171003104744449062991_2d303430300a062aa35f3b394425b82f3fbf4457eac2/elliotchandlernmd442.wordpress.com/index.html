Article response: “Reducing Cognitive Overload”

This post is a response to this article (IA).

 

This article provides ways to create user experiences that help the user be able to work with complex products. A failing in this zone is what led to the Discord Web communication app confusing me when I tried to use it it simply was too visually cluttered to be practical to use. There are a number of elements of design that the article suggests contribute to cognitive overload. One of the points that really struck a chord with me is the idea of simplicity not necessarily directly corresponding to the ease of use. The article gives

the screen shot to the right as an example of a user interface that, despite being simple, clean, and elegant, creates extreme cognitive overload for any user uninitiated into the arcane rites of its use. A reason this specifically resonates with me is my experiences with some similarly confusing computer user interfaces, most significantly apps created for the GNOME 3 desktop environment. For example, the Disks app,

shown to the left, while being simple, suffers from the same “mystery meat navigation” problem that afflicts the example shown in the article. While it is probably less intimidating to users who are entirely unfamiliar with computers, its tendency to break tradition from traditional computing metaphors makes it truly infuriating to use when one has familiarity with existing software, and the conventional meanings of icons. For instance, while the button, which looks like a “Stop” button from a cassette player, is logical in its meaning (it means “unmount”), it is confusing because it doesn’t follow the established convention for icons representing “unmount”, which is to use (an “eject” symbol), used by the other major GUIs (KDE, macOS, and Windows). One reason for doing this is that when a partition is unmounted, it remains available in the interface, with the button changing to (a “play” symbol), but this is not immediately obvious. A better solution would be to use a text label reading “Unmount” and changing to “Mount”, rather than a non-obvious icon. (Adding to the problem, the GNOME 3 Human Interface Guidelines suggest (IA) “[using] configuration options sparingly”, an approach which has been taken to an unnecessary extreme if the Guidelines’ exhortation to “Use progressive disclosure” were applied, configuration options would not place any significant burden on the software’s users.) Compare

the equivalent tool in an earlier version of macOS, which manages to be readable and fairly obvious in its operation, while being more visually elegant than the GNOME 3 version (although that last is rather subjective). (More recent versions of macOS have brought the interface progressively closer to the painful GNOME 3 version, which is why an older version is shown here.) Similarly,

the KDE Partition Manager is also quite clear in its operation, although personally I would prefer something between the KDE and macOS strategies. In summary, poorly executed simplification can lead to disorienting user interfaces that lead to cognitive overload for the user.

Article Responses: 14 Sept. 2017

“Metrics Versus Experience”

This article discusses how to be effective at making metrics-driven choices for Web-driven products, and points out that being user-focused and metrics-driven are not necessarily incompatible. There are a few points in it I especially like. One of these is the idea of focusing on retention metrics rather than conversion metrics. The article uses a “funnel” as a metaphor for user conversion and retention, which I think highlights a problem I have with a lot of Web-driven products, namely that they seem to optimize for the first part of the funnel (getting the potential user interested in the product, cultivating brand awareness, etc.) and neglect the other parts of the process (ease of onboarding, and product value / “reason” for retention). This has the consequence that I often am interested in Web services on the basis of the “buzz” surrounding them (in other words, I make it into the top of the funnel), but get tossed aside almost immediately because the services didn’t optimize the experience for those later parts of the funnel.

“How Google Improves User Experience with the HEART Framework”

This article, following up on the previous one, discusses Google’s specific strategy for Web-driven product choices, using a combination of metrics, user testing, and user feedback. I’d like to quarrel with this one, though: while the overall strategy is good, neglecting the principles described in “Metrics Versus Experience” leaves me frustrated. Specifically, the article presents the modal dialog shown in the picture at the right as “an easy, non-intrusive way to entice users to download and start using the better product experience“. While that works in some cases, there are two major problems with it. First, it is obnoxious and reduces the current user experience, even though the user could probably guess there was an app without getting nagged about it. Second, individual apps often don’t improve experience: often as not, they’re just slow and annoying.

Thoughts on “What Is Design?”

This is a response to this video: https://www.youtube.com/watch?v=cOx_Zx95hxM.

 

This video is a talk from designer Bill Moggridge.

Unlike “What Is User Experience Design?”, this talk focuses on design in the more general sense, covering the field as a whole, rather than focusing only on Web design. One case study given in the talk that interested me was that of the successful i-mode cellular service from the 1990s. It shows a feature of the service drink purchase using the Internet that Moggridge points out is poorly designed. The problem with the service is that it is very slow, and most customers aren’t interested in waiting for it. Moggridge observes that many poor designs are connected to technology. Even so, the capabilities of technology systems are so compelling that users are willing to tolerate some bad design in order to take advantage of the substantial abilities of the technologies.

Thoughts on “What Is User Experience Design?”

This is a response to this article: https://www.smashingmagazine.com/2010/10/what-is-user-experience-design-overview-tools-and-resources/.

 

This article focuses on Web site design. It provides an overview of the user experience design profession, as it applies to Web site UX design. In addition, it provides a directory of resources for those interested in user experience design. It’s especially interesting to me how it provides many examples of Web sites that I would rather have be designed differently, from a graphic design perspective, but nevertheless have workflows that I can appreciate (e.g. Amazon): I get the impression that the distinction between the overall workflow and the actual design and content of the work being designed is important to getting user acceptance. In my case, when there’s a Web site that I want to use but that bothers me, a lot of the time the problems are either missing features, or things that can be removed using a Web content blocker such as uBlock Origin. From that, I think that the negative response I have to it is largely drawn from elements of the site outside of the actual workflow that the UX design process would have created, and instead is reacting to poor content overlaid onto an effectively designed workflow. The article suggests using A/B testing software, writing that “Split testing determines which version is better based on these statistics”. I think that supports that distinction between workflow and content, because the UX that A/B testing would be examining would generally be the workflow, and not the content. Consequently, the content and the “features” I mentioned earlier do not really get tested.
