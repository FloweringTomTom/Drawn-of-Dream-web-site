Article response: "Reducing Cognitive Overload"

This post is a response to this article (IA).

This article provides ways to create user experiences that help the user be able to work with complex products. A failing in this zone is what led to the “Discord” Web communication app confusing me when I tried to use it — it simply was too visually cluttered to be practical to use. There are a number of elements of design that the article suggests contribute to cognitive overload. One of the points that really struck a chord with me is the idea of simplicity not necessarily directly corresponding to the ease of use. The article gives

the screen shot to the right as an example of a user interface that, despite being simple, clean, and elegant, creates extreme cognitive overload for any user uninitiated into the arcane rites of its use. A reason this specifically resonates with me is my experiences with some similarly confusing computer user interfaces, most significantly apps created for the GNOME 3 desktop environment. For example, the Disks app, [caption id="attachment_178" align="alignnone" width="300"] Disks from GNOME 3 (source [IA])[/caption]shown to the left, while being simple, suffers from the same "mystery meat navigation" problem that afflicts the example shown in the article. While it is probably less intimidating to users who are entirely unfamiliar with computers, its tendency to break tradition from traditional computing metaphors makes it truly infuriating to use when one has familiarity with existing software, and the conventional meanings of icons. For instance, while the ⏹️︎ button, which looks like a "Stop" button from a cassette player, is logical in its meaning (it means "unmount"), it is confusing because it doesn't follow the established convention for icons representing "unmount", which is to use ⏏︎ (an "eject" symbol), used by the other major GUIs (KDE, macOS, and Windows). One reason for doing this is that when a partition is unmounted, it remains available in the interface, with the button changing to ▶︎ (a "play" symbol), but this is not immediately obvious. A better solution would be to use a text label reading "Unmount" and changing to "Mount", rather than a non-obvious icon. (Adding to the problem, the GNOME 3 Human Interface Guidelines suggest (IA) "[using] configuration options sparingly", an approach which has been taken to an unnecessary extreme — if the Guidelines' exhortation to "Use progressive disclosure" were applied, configuration options would not place any significant burden on the software's users.) Compare [caption id="attachment_220" align="alignright" width="300"] Partition management in macOS (source [IA])[/caption]the equivalent tool in an earlier version of macOS, which manages to be readable and fairly obvious in its operation, while being more visually elegant than the GNOME 3 version (although that last is rather subjective). (More recent versions of macOS have brought the interface progressively closer to the painful GNOME 3 version, which is why an older version is shown here.) Similarly, [caption id="attachment_237" align="alignleft" width="300"] KDE Partition Manager (source [IA])[/caption]the KDE Partition Manager is also quite clear in its operation, although personally I would prefer something between the KDE and macOS strategies. In summary, poorly executed simplification can lead to disorienting user interfaces that lead to cognitive overload for the user.
